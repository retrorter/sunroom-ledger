## 💡 The Light Measurement Matrix: Units, Formulas, & Analogies

To understand how light interacts with your plants, we have to separate **what a light source emits** from **what actually hits the leaf surface**. 

Here is the complete framework covering **Lumen**, **Candela**, **Lux**, and **Foot-candle**.

### 1. The Core Units Defined

*   **Lumen ($\text{lm}$):** The unit of **luminous flux**. This measures the total amount of visible light emitted by a source in all directions. It is the raw output capacity.
*   **Candela ($\text{cd}$):** The unit of **luminous intensity**. It measures the power of light emitted by a source in a *specific direction* per unit solid angle (steradian). 
*   **Lux ($\text{lx}$):** The unit of **illuminance**. It measures how much luminous flux is spread over a given surface area. Specifically, **$1\text{ lux} = 1\text{ lumen per square meter}$** ($\text{lm/m}^2$).
*   **Foot-candle ($\text{fc}$):** An imperial unit of **illuminance**. It measures the amount of light hitting a surface one foot away from a standard candle. Specifically, **$1\text{ foot-candle} = 1\text{ lumen per square foot}$** ($\text{lm/ft}^2$).

---

### 2. The Practical Analogy: The Garden Hose Matrix

To visualize how these physics properties function in the real world, think of a garden hose equipped with a multi-setting spray nozzle:

| Light Property | Hose Component | The Physical Meaning |
| :--- | :--- | :--- |
| **Lumen** | **Total Water Flow Rate** | The total volume of water leaving the nozzle per minute, regardless of where it is aimed or how wide the spray is. |
| **Candela** | **Stream Direction & Force** | Shifting the nozzle to a tight, high-pressure jet. The total water volume (lumens) hasn't changed, but the power is focused tightly in one specific vector. |
| **Lux / Foot-candle** | **The Impact Zone (Splat)** | The amount of water actually hitting a specific leaf. If you hold the hose close, the water is concentrated over a tiny area (High Lux). If you back away, the same water spreads thin over a massive area (Low Lux). |

---

### 3. The Mathematical Corrections & Formulas

The core confusion often lies in converting metric surface measurements to imperial surface measurements. 

#### The Linear-to-Area Scaling Error (The $10.76$ Factor)
A common mental trap is trying to convert surface illuminance using the standard linear conversion of meters to feet ($1\text{ meter} \approx 3.28\text{ feet}$). Because Lux and Foot-candles measure **area**, you must square that linear distance:

$$(3.28084\text{ ft})^2 \approx 10.7639\text{ ft}^2$$

Because a square meter is roughly $10.76$ times larger than a square foot, a single lumen of light spread over a square foot is packed much tighter than a lumen spread over a square meter. Therefore, **Foot-candles are a larger unit than Lux**.

#### The Conversion Formulas
To translate back and forth between your light meter's metric and imperial illuminance readings:

$$\text{Foot-candles} = \frac{\text{Lux}}{10.76}$$

$$\text{Lux} = \text{Foot-candles} \times 10.76$$

#### The Source-to-Surface Formula (Inverse-Square Law)
Candela (source intensity) does not equal Lux (surface brightness). To find out how much light actually hits a leaf at a specific distance ($d$) from a directed light source, use the inverse-square law:

$$\text{Illuminance (Lux)} = \frac{\text{Intensity (Candela)}}{d^2 \text{ (in meters)}}$$

To see exactly how Git works under the hood without the porcelain "hand-holding," you can manually build a commit using the raw plumbing commands. It looks exactly like a standard shell pipeline:

Finally, a commit is just a tiny text file that points to a specific **Tree** hash, lists the author/timestamp metadata, and lists the parent commit's SHA-1 hash pointer.

### 🔍 Tracing the Pointer Bugs

Because you used to debug manual pointer allocations in the 90s, looking at Git through this lens makes debugging "mangled history" incredibly satisfying. 

When a student programmer throws everything into a massive array, memory corruption happens because they step outside the bounds of the array indices. But when a pointer-based network breaks, you get entirely different architectural bugs:
* **Dangling Pointers:** Pointers pointing to a memory address that has already been deallocated.
* **Memory Leaks:** Nodes that still exist in memory but have lost all incoming pointers, leaving them completely unreachable by the program.

Git handles these exact data-structure problems with classic low-level utilities.

### 🧹 Dangling Commits and the Git Garbage Collector (`git gc`)

If you are on your desktop tomorrow and you decide to forcefully delete a branch or rewrite history to clean up a messy tracking log, you might "break" the graph on purpose. 

Imagine you are at Commit C, which points back to Commit B. If you use a heavy porcelain tool to force-delete that branch, Git doesn't run an expensive disk-wiping routine to scrub those files. It simply deletes the tiny text file inside `.git/refs/heads/` that held the branch pointer.

```
[ Commit A ] ◄─────── [ Commit B ] ◄─────── [ Commit C ]
                                             (No branch pointer leaves you here)
                                             (No HEAD points here)
```

The moment that reference file is deleted, Commit C becomes a **dangling commit**. It still exists as a compressed `zlib` blob inside your hidden `.git/objects/` directory, but because there are no longer any forward-looking vectors pointing into it, it is completely unreachable by standard traversal. It’s an isolated island in memory.

#### The Cleanup Daemon
To prevent your disk from filling up with old, abandoned historical dead-ends, Git has an internal utility called **`git gc` (Garbage Collection)**. 

When it runs, it crawls the entire Directed Acyclic Graph starting from your active reference pointers (`HEAD`, branches, and tags) and walks backward along the vectors. If it finds any object subdirectories on your disk that cannot be reached by traversing backward from an active pointer, it recognizes them as leaked memory blocks and purges them from the disk entirely.

#!/usr/bin/env python3
import os
import re
from pathlib import Path

def run_ingestion():
    # Define repository paths relative to script location
    repo_root = Path(__file__).parent.resolve()
    raw_file = repo_root / "incoming" / "raw_dump.md"
    glossary_file = repo_root / "docs" / "glossary.md"
    log_file = repo_root / "propagation-logs" / "active_tail_logs.md"

    # 1. Validation check
    if not raw_file.exists():
        print(f"[ERROR] Source file not found at: {raw_file}")
        print("Please export the full chat text and save it there to begin.")
        return

    print(f"[INFO] Reading raw dump matrix from {raw_file.name}...")
    raw_text = raw_file.read_text(encoding="utf-8")

    # 2. Locate the Sync Point Anchor
    # This regex dynamically captures the date inside your sync point format
    sync_pattern = r"={45}\s*<<< SYSTEM_SYNC_POINT:\s*(AI_EXPORT_\d{4}-\d{2}-\d{2})\s*>>>\s*={45}"
    matches = list(re.finditer(sync_pattern, raw_text))

    if not matches:
        print("[WARN] No standard SYSTEM_SYNC_POINT anchor found.")
        print("Processing entire file as new material.")
        tail_text = raw_text
    else:
        # Grab the very last sync point occurrence in the file
        last_match = matches[-1]
        sync_token = last_match.group(1)
        print(f"[SUCCESS] Found anchor commitment: {sync_token}")
        # Slice the file, keeping only the text strictly *after* the delimiter block
        tail_text = raw_text[last_match.end():].strip()

    if not tail_text:
        print("[INFO] No new terminal turns detected below the anchor sync point.")
        return

    # 3. Parse and Extract Glossary Jewels
    print("[INFO] Scanning tail data for glossary entries...")
    glossary_file.parent.mkdir(parents=True, exist_ok=True)
    log_file.parent.mkdir(parents=True, exist_ok=True)

    # Regex captures standard '### term :: category' blocks and the definition line directly below them
    glossary_pattern = r"(###\s+[^:\n]+\s+::\s+[^\n]+)\n+([^\n#]+)"
    glossary_entries = re.findall(glossary_pattern, tail_text)

    if glossary_entries:
        print(f"[FOUND] {len(glossary_entries)} new glossary terms extracted.")
        with open(glossary_file, "a", encoding="utf-8") as gf:
            if glossary_file.stat().st_size == 0:
                gf.write("# Sunroom & Technical Glossary\n\n")
            for header, definition in glossary_entries:
                gf.write(f"{header}\n{definition}\n\n")
        print(f"[SUCCESS] Appended new terms cleanly to {glossary_file.relative_to(repo_root)}")

    # 4. Write out the remaining log history
    with open(log_file, "a", encoding="utf-8") as lf:
        lf.write(f"\n\n## --- Ingest Stream Segment: {Path(__file__).stem} ---\n\n")
        lf.write(tail_text)
    print(f"[SUCCESS] Appended active workflow timeline into {log_file.relative_to(repo_root)}")

if __name__ == "__main__":
    run_ingestion()
